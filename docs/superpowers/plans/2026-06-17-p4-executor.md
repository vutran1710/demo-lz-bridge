# P4 — Executor Worker Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Build the horizontally-scalable Executor worker (commit + deliver) with an on-chain ExecutorConfig, refactor the DVN to verify-only, and prove it with sad-path/failure e2e suites — replacing the harness-as-executor.

**Architecture:** DVN attests (`verify` only); Executor owns `commitVerification` + `lzReceive`. Executor is a Go worker with separated stages (watch → commit → schedule → signer-pool → execute → retry) using a **channel-sharded signer pool** (per-channel nonce order, parallel across channels). On-chain `ExecutorConfig` supplies the gas budget; `lzReceive` enforces it.

**Tech Stack:** Solidity/Foundry, Go/go-ethereum, TS+Vitest+viem, Anvil. **Branch each milestone fresh from `main`; one PR per milestone; merge before next.** (See [[milestone-branching-from-main]].)

**Build discipline:** sad-path e2e suites authored RED before the worker logic (test-suites-first); TDD unit tier (`forge test` / `go test`) per implementation step.

## Multi-chain support (design — the real scalability axis)

Multi-chain is **not** a new protocol primitive: an N-chain network is just **multiple bilateral A↔B pathways**, each being a per-chain contract set (Endpoint + libs + ExecutorConfig) plus a `(UlnConfig, ExecutorConfig)` for that directed pathway. The protocol already handles "a pathway"; scaling to N chains = replicating it.

The requirement this places on the **workers** (DVN/attestor and Executor): each must be **multi-pathway** — configured with a **list** of pathways, not a single `SRC/DST`. So:
- **Worker config generalizes** from `{SRC_RPC, DST_RPC, SRC_ENDPOINT, DST_RECEIVE_LIB}` to a list of pathway descriptors `[{srcRpc, srcEndpoint, dstRpc, dstReceiveLib, dstEndpoint, executorConfig, confirmations}]`. The worker dials each chain once (dedup RPCs) and runs the watch→verify (DVN) / watch→commit→deliver (Executor) loop **per pathway**, sharing the signer pool.
- **A DVN operator supports many chains** by listing the pathways it serves; same binary.
- The **3-chain ring e2e** (`A→B→C→A`) is the validation: 3 pathways serviced by the same DVN + Executor, with a `RelayApp` that forwards each hop; we assert `Received` on B, then C, then back on A.

This is implemented in **P4.M3** (worker becomes multi-pathway) and validated in **P4.M4** (ring e2e). It is the concrete answer to "is the protocol horizontally scalable": add chains → add pathways → the same workers service them.

```solidity
// SPDX-License-Identifier: MIT
pragma solidity ^0.8.24;
import "./interfaces/IExecutor.sol";
import "./interfaces/IEndpoint.sol"; // SetConfigParam

contract ExecutorConfig {
    mapping(address => mapping(uint32 => ExecutorConfigData)) internal _cfg; // oapp => dstEid => cfg
    function setConfig(address oapp, SetConfigParam[] calldata params) external {
        for (uint256 i = 0; i < params.length; i++) {
            if (params[i].configType == CONFIG_TYPE_EXECUTOR) {
                _cfg[oapp][params[i].eid] = abi.decode(params[i].config, (ExecutorConfigData));
            }
        }
    }
    function getConfig(address oapp, uint32 dstEid) external view returns (ExecutorConfigData memory) {
        return _cfg[oapp][dstEid];
    }
}
```
