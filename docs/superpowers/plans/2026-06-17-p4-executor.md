# P4 — Executor Worker Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax.

**Goal:** Build the horizontally-scalable, **multi-chain** Executor (commit + deliver) with an on-chain ExecutorConfig, refactor the DVN to verify-only, and prove robustness with sad-path/failure e2e + a 3-chain ring — replacing the harness shims.

**Architecture:** DVN attests (`verify` only); Executor owns `commitVerification` + `lzReceive`. The Executor is a Go worker with separated stages (watch → commit → schedule → signer-pool → deliver) using a **channel-sharded signer pool** (per-channel nonce order, parallel across channels). Both DVN and Executor are **multi-chain**: one process takes a **pathway-list**, dials each unique chain once, shares a per-destination signer pool, and uses one DVN key/address across chains.

**Tech Stack:** Solidity/Foundry, Go/go-ethereum, TS+Vitest+viem, Anvil. **Branch each milestone fresh from `main`; one PR per milestone; merge before next** (see [[milestone-branching-from-main]]).

**Build discipline:** sad-path/e2e suites authored RED before worker logic; TDD unit tier (`forge test` / `go test`) per step; every e2e test has an explicit bounded timeout so a stuck test fails fast.

## Multi-chain support (design — verified against LayerZero docs)

**On-chain (already implemented):** one immutable Endpoint per chain with an **eid-keyed registry**; the sender passes `dstEid`. Our `Endpoint` already does this (`sendLib[oapp][eid]`, `receiveLib[oapp][srcEid]`, `uln[oapp][srcEid]`, `ExecutorConfig[oapp][dstEid]`). Adding a chain = deploy the contract set + configure new eid pathways. No per-pair contracts.

**Off-chain (the work, M5):** a DVN/Executor is **multi-chain** — config is a list of pathway descriptors `[{srcRpc, srcEndpoint, dstRpc, dstReceiveLib, dstEndpoint, executorConfig, confirmations}]`. The worker dials each unique chain once and runs its per-pathway loop; **same DVN key/address across chains**; **per-destination signer pool**. Validated by the **3-chain ring** (M6).

## M4 — Sad-path e2e (single pathway) `[branch: milestone/p4-exec-sadpaths]`

Author each RED, then make green. Every test has an explicit bounded timeout.

### Tasks M4.1–M4.6 (one suite each)
- [ ] **executor-down**: DVNs verify; no executor ⇒ not delivered within 15s (assert no `Echoed`); start executor ⇒ delivered. (timeout 40s)
- [ ] **below-threshold**: 1 of 3 DVNs (M=2) + executor ⇒ never committed/delivered within 15s (fail-safe). (timeout 30s)
- [ ] **revert-retry**: `AppRevert` ⇒ parked (committed, not delivered); `setFailing(false)` ⇒ executor retry delivers; nonce not skipped. (timeout 40s)
- [ ] **signer-crash**: executor with 2-signer pool under burst; kill one signer ⇒ all delivered. (timeout 60s)
- [ ] **ordering**: burst on one channel ⇒ `Echoed` in strict nonce order. (timeout 60s)
- [ ] **idempotent-restart**: restart executor mid-flight ⇒ no double-`Echoed`, no gaps. (timeout 60s)
- [ ] Each: write RED → fix worker until green → commit. **Open PR (M4).**

**CA gate M4:** all 6 sad-path suites green. PR → main, merge.
